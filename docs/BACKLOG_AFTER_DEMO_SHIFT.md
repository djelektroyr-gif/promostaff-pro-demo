# Бэклог после демо-смены (PRO Demo)

**Когда брать в работу:** после завершения текущей тестовой смены в `promostaff-pro-demo`, чтобы не мешать приёмке сценария.

---

## 1. Заказчик не должен видеть ставки и внутренние суммы

**Цель:** ставка ₽/час и расчёты к оплате / итоги в ₽ — только для администратора и исполнителя (по продуктовым правилам); в интерфейсе заказчика их не показывать.

**Где смотреть (ориентиры по коду):**

- `handlers/shifts.py` — строки с `💰 Ставка`, блоки с `к оплате`, `ИТОГО` (карточки смены / отчёты при просмотре клиентом).
- `services/shift_hub.py` — строка «Ставка» в тексте хаба смены.
- При необходимости: `handlers/shift_center.py`, `handlers/admin.py` (админ оставляем как есть).

**Реализация:** ветвление по роли (`get_client` / `is_admin_user`) или отдельные шаблоны текста для клиента без полей `rate` и без денежных итогов.

---

## 2. Уведомления заказчику не должны дублировать «администраторский» текст

**Проблема:** в ряде мест заказчику уходит тот же текст, что и администраторам (операционный тон, внутренние формулировки).

**Где править:**

- `services/shift_notifier.py` — для каждого блока, где после цикла по `ADMIN_USER_IDS` отправляется сообщение на `client_id`, завести **отдельную строку** для клиента: короче, без дублирования служебных инструкций («свяжитесь с исполнителем…» можно смягчить или переформулировать под заказчика).
- `handlers/tasks.py` — функция `_notify_client_task_completed`: клиент получает развёрнутое сообщение; админам уходит `send_all_admins` с кратким текстом — убедиться, что заказчик **не** получает второй раз тот же поток, что админы (если заказчик одновременно в `ADMIN_USER_IDS`, продумать одно сообщение или разные каналы).

**Критерий готовности:** заказчик видит понятные статусные уведомления без «внутренней кухни» и без лишнего совпадения дословно с текстом для админа.

**Краевой случай:** если `client_id` совпадает с одним из `ADMIN_USER_IDS`, при текущей логике возможен **двойной** push одного и того же события — при рефакторинге предусмотреть дедупликацию или одно объединённое сообщение.

---

## 3. Отчёт по PROMOSTAFF PRO (основной проект)

**Цель:** подготовить **сводный отчёт** по согласованному с заказчиком формату по основному репозиторию `promostaff-bot` (реализованные доработки, статус проверки, открытые хвосты).

**Куда положить:** разумно разместить в `promostaff-bot/docs/` (например, отдельный файл вида `REPORT_PRO_IMPLEMENTATION_<дата>.md` или обновить существующий мастер-док, если так договорились).

**Связь с демо:** в отчёте можно кратко отметить, что проверено в `promostaff-pro-demo` и что переносится/не переносится в прод-бота.
